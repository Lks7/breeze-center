## 后端搭建 — Go + chi + TOML 配置加载

### 技术选型决策
- 后端框架：**chi**（不是 gin）。理由：路由风格贴近 net/http，中间件生态够用，个人项目不需要 gin 的魔法
- 配置解析：BurntSushi/toml
- CORS：rs/cors
- 数据库：SQLite（暂未启用，当前阶段配置即数据）
- 模块路径：`github.com/breeze/center`

### 已完成
1. Go module 初始化与目录骨架
   - `server/cmd/server/main.go` — 入口
   - `server/internal/config/` — TOML 类型与加载器
   - `server/internal/handler/` — health/services/widgets handler
   - `server/internal/middleware/` — CORS + 请求日志
   - `server/internal/model/` — API 响应模型
   - `server/internal/service/` — 服务注册表（内存查询）
2. 配置加载与继承合并机制（核心设计落地）
   - `config/site.toml` — 全局站点配置（端口、CORS origins）
   - `config/services/*.toml` — 服务通用模板（6 个：blog/github/rss-reader/bookmark/service-status/todo）
   - `config/providers/personal/*.toml` — 个人实例（4 个，继承 base_service）
   - 合并规则：模板字段为默认值，instance 直接挂载，override.widget 覆盖模板 widget
3. HTTP API 实现
   - `GET /api/health` — 健康检查
   - `GET /api/v1/services` — 服务列表（支持 ?category= 过滤）
   - `GET /api/v1/services/categories` — 分类分组
   - `GET /api/v1/services/{id}` — 单个服务详情
   - `GET /api/v1/widgets` — 所有 widget 配置
   - `GET /api/v1/widgets/data?type=xxx` — widget 数据（mock，按 type 返回）
4. JSON 字段统一 snake_case（struct 加 json tag）
5. Vite 代理配置：`/api` → `http://localhost:8080`

### 启动方式

## 后端管理后台 — SQLite + 动态 CRUD + 管理界面

### 核心设计纠正
breeze 指出：之前数据全死在 TOML/mock 里，不叫真正的后端。重新定下两层模型：
- **TOML（保留）** — 服务**类型**模板（blog/rss-reader 是什么、widget 类型），Git 编辑
- **SQLite（新增）** — **动态内容**（博客文章、RSS 源、书签、待办、服务实例），管理后台 CRUD

### 数据层
- 用 `modernc.org/sqlite`（纯 Go，无 cgo），Windows 编译零负担
- `server/internal/store/` 一实体一文件：`blog.go` / `rss.go` / `bookmark.go` / `todo.go` / `service.go`
- schema 自动迁移（CREATE TABLE IF NOT EXISTS）
- 主键统一 TEXT UUID；时间字段 TEXT ISO8601；软删除 deleted_at
- 启动时 `--data ../data`，DB 落到 `data/breeze.db`

### Admin CRUD API（/api/v1/admin/*）
- blog/posts — List/Get/Create/Update/Delete
- rss/sources — List/Create/Update/Delete；rss/articles — List/MarkRead
- bookmarks — 全 CRUD
- todos — 全 CRUD + `PATCH /{id}/toggle` 快速切换完成
- services — 全 CRUD（动态实例，与 TOML 模板并存）

### 前端管理后台（/admin）
- React Router 加 BrowserRouter，首页 `/` 与管理 `/admin` 分流
- `AdminLayout` 左侧侧边栏 + 右侧 Outlet，复用玻璃卡片设计系统
- 通用 `AdminPage<T>` 组件（fields 配置驱动列表+表单弹层），5 个实体复用同一套骨架
- `TodoAdmin` 是单独的快速添加式交互（不套通用组件）
- API client 在 `web/src/api/`：`client.ts`（fetch 封装）+ `admin.ts`（各实体调用）

### 启动方式（更新）
- 后端：`cd server && ./bin/server.exe --config ../config --data ../data`
- 前端：`cd web && pnpm dev`
- 访问 `http://localhost:4321/admin` 进管理后台

## 创建用户级 skill：ponytail-lazy-coder

breeze 让我把 https://github.com/DietrichGebert/ponytail 这个仓库"学会"。
仓库本身是 AI 编码插件（支持 16+ 代理），核心理念是"让 AI 像最懒的高级开发者一样思考"。

我把它的方法论提炼成 WorkBuddy 用户级 skill：
- 位置：`~/.workbuddy/skills/ponytail-lazy-coder/SKILL.md`
- 形态：纯方法论 markdown（无 scripts/assets，不需要）
- 核心：**Lazy Ladder** 7 级阶梯——写代码前依次问：是否存在 / 代码库已有 / 标准库 / 平台原生 / 已装依赖 / 一行搞定 / 最后才写最小代码
- 关键原则：对解决方案懒惰，对阅读绝不懒惰（验证/错误处理/安全/可访问性永远不砍）
- 三档强度：lite（1-3 级）/ full（全部，默认）/ ultra（全部+反抽象）
- 用 `skill-creator` 的 init_skill.py 初始化，quick_validate.py 验证通过
- 注意：Windows 下验证脚本要 `PYTHONUTF8=1` 才能读 UTF-8 SKILL.md

后续写代码会自动按这套规则走。
