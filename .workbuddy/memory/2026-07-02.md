## 基金盈亏功能 v1.1 重写（FUND_FEATURE）

### 起因
用户要求根据 docs/FUND_FEATURE.md 完善基金盈亏功能让它"实用"。
审查代码后发现 v1.0 存在三个致命问题：
1. **数据源是假设的**：必应 API `api.biyingapi.com/fd/info/{code}/{licence}` 端点未经证实、需付费 licence
2. **盈亏计算 Bug**：handler 把盈亏写回 BuyAmount 字段（`holdings[i].BuyAmount = profit`），导致前端无法同时显示购买金额和盈亏
3. **前端用 any 类型**：fundAPI 没有类型定义，UI 仅套用通用 AdminPage 只能显示一行摘要

### 完成内容
- **后端 fetcher 重写**：改用天天基金免费公开接口 `http://fundgz.1234567.com.cn/js/{code}.js`（JSONP 格式），解析出 dwjz/gsz/gszzl/jzrq/name，无需任何 API Key
- **后端 store 扩展**：新增 UpdateHolding 方法（修改购买信息后重算 shares）
- **后端 handler 重写**：引入 FundHoldingView DTO 隔离原始字段与计算字段（current_value/profit/profit_rate），新增 PUT 修改、单条更新净值、总盈亏统计 3 个接口
- **前端类型强化**：types/entities.ts 新增 FundHolding/FundHoldingInput/FundSummary/FundNavUpdateResponse/FundNavUpdateResult 五个类型，fundAPI 全部强类型化
- **前端页面重写**：手写独立 FundAdmin.tsx，4 张统计卡片 + 完整表格 + 单条更新 + 红涨绿跌（中文市场约定，盈利 #dc2626 / 亏损 #16a34a）

### 关键技术决策
- **不引入 golang.org/x/text**：天天基金接口实际返回 UTF-8（不是 GBK），用 hexdump 验证字节 `e5 8d 8e` = "华"
- **去掉 settings.fund_api_licence 依赖**：数据源免费
- **不存 nav_date 字段**：保持 schema 不变，last_updated 已够用
- **单条更新净值会同步回写真实基金名称**：用户创建时 name 可留空

### 端到端验证
- 后端 go build 通过
- 前端 tsc --noEmit 零错误
- 实测 000001 华夏成长混合：当前净值 1.596，盈亏 +640 元，收益率 +6.4%
- 数据库存储 UTF-8 字节正确，HTTP 响应字节正确

### 调试踩坑
Windows Git Bash 的 curl | python 管道会损坏 UTF-8 字节（被 GBK 解码再编码），导致 JSON 显示 surrogate 乱码。直接用 Python urllib 或查数据库验证字节正确。

## 修复删除持仓无响应

### 根因
后端 `DeleteHolding` 返回 `204 No Content`（无响应体），但前端通用 `request()` 函数强制 `res.json()` 解析。204 没有正文 → JSON 解析失败 → fallback 成 `{success:false}` → 抛 APIError。mutation 只有 `onSuccess` 没有 `onError`，所以对话框不关闭，看起来"点击没反应"。

### 修复
把 fund 的 DeleteHolding 从 `w.WriteHeader(204)` 改成 `writeData(w, 200, {"deleted":true})`，与项目内 fusion/rss 等删除接口对齐。未改前端通用 `request()`，避免影响其他接口。

### 验证
curl 实测：POST 创建 → DELETE 返回 `{"data":{"deleted":true},"success":true}` HTTP 200。

### 教训
- 项目内所有删除接口应统一返回 200 + JSON body，不要用 204（前端 request 强制解析 JSON）
- mutation 应加 `onError` 至少做日志/提示，否则失败时 UI 无反馈

## 基金功能 v1.2：净值历史 + 定时抓取 + 止盈止损预警

### 功能 1：净值历史 + 走势曲线
- 新表 `fund_nav_history`（id, holding_id, code, nav, recorded_at）
- store.UpdateCurrentNav 每次抓取自动写一条历史快照
- handler 新增 `GET /admin/fund/holdings/{id}/history` + `GET /admin/fund/history`
- 前端两张新图：
  - **NavTrendChart**：单只基金净值历史走势折线图（持仓行点击展开，渐变填充 + 买入净值参考线）
  - **CumulativeProfitChart**：整体累计收益曲线（按时间聚合各持仓快照，盈利红填充/亏损绿填充）

### 功能 2：定时自动抓取
- 新文件 `server/internal/fetcher/fund_scheduler.go`
- 每分钟检查一次，交易日（周一~周五）15:30 触发抓取
- 周末跳过，启动时如果当天 15:30 已过且未抓过则补抓一次
- alreadyFetchedToday 通过 last_updated 日期判断避免重复
- 抓取后同步写历史 + 检查预警（复用 checkAlertForHolding）
- main.go 注册：`fundScheduler := fetcher.NewFundScheduler(fundStore, notifStore)`

### 功能 4：止盈止损预警
- fund_holdings 表新增字段：`target_profit_rate`、`stop_loss_rate`、`alert_triggered`
- schema.go 增量迁移：ALTER TABLE ADD COLUMN（已存在列时忽略 duplicate column error）
- handler.checkAlert：抓取净值后检查盈亏率是否越过阈值
  - 止盈：profitRate >= target_profit_rate（正值，如 0.2=20%）
  - 止损：profitRate <= stop_loss_rate（用户可输正数表示"跌X%"，内部转负）
  - 触发后推 NotificationStore.Push（type="fund"）+ MarkAlertTriggered 防重复
- 前端编辑表单加止盈率/止损率输入（百分比显示，提交时 ÷100 转小数）
- 详情页加预警设置总览卡片 + 持仓行预警铃铛图标

### 端到端验证
- 后端 go build 零错误，前端 tsc 零错误
- 实测：设 target_profit_rate=0.001（0.1%）→ 抓取净值 → 收益率 26.38% → alert_triggered=true → 通知推送成功
- 历史记录写入正常，API 返回数据结构正确
- 重置为合理值：止盈 20%，止损 10%

### 关键设计决策
- **store.UpdateCurrentNav 同时写历史**：调用方无需关心历史记录，自动留痕
- **alert_triggered 防重复**：一轮触发后不再推通知，直到用户修改设置才重置
- **止损率允许正数输入**：用户输 10 表示"跌 10%"，内部统一转 -0.1，降低心智负担
- **调度器用 1 分钟轮询而非 cron**：复用 RSS scheduler 的 ticker 模式，保持代码风格一致
- **节假日不处理**：周末跳过，节假日数据会在下一个交易日补上（last_updated 判断）
