# 学习并应用 Ponytail 开发哲学

## Goal

学习并内化 Ponytail 开发哲学（"懒惰的资深开发者"方法论），将其核心原则整合到项目的开发规范和 AI 辅助开发流程中。

## Background

Ponytail 是一个代码质量技能，通过"写代码前先走完决策阶梯"的方法论，实测减少 54% 代码量、20% 成本、27% 时间，同时保持 100% 安全性。

核心理念：在写代码之前，按优先级检查以下阶梯：
1. 这个功能需要存在吗？ → 不需要就跳过 (YAGNI)
2. 代码库里已经有了？ → 复用，别重写
3. 标准库能做？ → 用标准库
4. 原生平台特性？ → 用原生特性
5. 已安装的依赖？ → 用现有依赖
6. 一行能搞定？ → 就写一行
7. 只有到这一步才写最小可工作代码

**永不妥协的底线**：验证、错误处理、安全、可访问性 — 这些永远不能省略。

## Requirements

### 1. 文档化 Ponytail 原则
- 将 Ponytail 的决策阶梯和核心原则整理成中文文档
- 记录典型案例和反模式示例
- 明确"永不妥协"的质量底线

### 2. 整合到项目规范
- 将 Ponytail 原则写入 `.trellis/spec/` 的相关层级规范
- 为 backend 和 frontend 各创建适配版本
- 确保规范可以被 AI 助手在开发时自动注入

### 3. 实践验证
- 在实际代码审查中应用 Ponytail 检查清单
- 识别项目中现有的过度工程案例
- 记录应用效果和改进建议

### 4. 建立审查机制
- 创建 Ponytail 风格的代码审查检查清单
- 集成到 `.trellis/spec` 的质量检查流程
- 可选：创建自动化审查脚本或 skill

## Out of Scope

- 不需要安装 Ponytail 插件到特定 IDE（我们关注原则本身）
- 不需要重构现有代码库（仅应用于新代码）
- 不需要创建完整的自动化工具（可以是未来任务）

## Notes

这是一个学习型任务，重点在于理解和内化方法论，而非工具安装。核心价值是：
- **懒于方案，勤于阅读**：先理解问题和现有代码，再选择最简方案
- **代码越少，维护越简单**：但不是 code golf，而是只写必要的代码
- **质量底线不可妥协**：安全、验证、错误处理、可访问性永远不能省
