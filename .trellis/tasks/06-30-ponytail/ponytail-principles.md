# Ponytail 开发哲学

## 核心理念

**"懒惰的资深开发者"** —— 写代码之前先想清楚是否真的需要写。

代码越少，维护成本越低、bug 越少、理解越容易。但这不是 code golf（代码高尔夫），而是**只写必要的代码**。

## 决策阶梯

在写任何代码之前，按顺序检查以下阶梯，在第一个匹配的阶梯停下：

### 1. 这个功能需要存在吗？
**YAGNI 原则**（You Aren't Gonna Need It）

- ❌ **过度设计**：为"可能的未来需求"添加抽象层、配置项、插件系统
- ✅ **正确做法**：只实现当前明确需要的功能
- 💡 **示例**：不要为"可能需要支持多种格式"而设计通用导出器，先实现确定需要的 CSV 导出

### 2. 代码库里已经有了？
**复用优于重写**

- ❌ **常见错误**：没搜索就写新函数，创建功能重复的工具类
- ✅ **正确做法**：先搜索现有代码，理解现有模式，扩展而非重写
- 💡 **示例**：需要格式化日期前，先检查项目是否已有日期工具函数

### 3. 标准库能做？
**优先使用语言内置功能**

- ❌ **过度依赖**：为简单操作引入第三方库
- ✅ **正确做法**：熟悉语言标准库的能力
- 💡 **示例**：
  - JavaScript: 不需要 lodash 来做数组去重，用 `[...new Set(arr)]`
  - Python: 不需要额外库来解析 JSON，用内置 `json` 模块

### 4. 原生平台特性？
**利用浏览器/OS/运行时的原生能力**

- ❌ **重复造轮子**：安装库来实现平台已提供的功能
- ✅ **正确做法**：了解平台能力，直接使用
- 💡 **经典案例**：
  - **日期选择器**：不需要 flatpickr，用 `<input type="date">`
  - **颜色选择器**：不需要组件库，用 `<input type="color">`
  - **模态框**：不需要 React Modal，用 `<dialog>` 元素
  - **表单验证**：不需要库，用 HTML5 `required`, `pattern`, `min`, `max`

### 5. 已安装的依赖？
**利用项目现有依赖**

- ❌ **依赖膨胀**：为小功能引入新的大型依赖
- ✅ **正确做法**：查看 `package.json` / `requirements.txt`，用现有的
- 💡 **示例**：项目已有 axios，不要为一个请求引入 fetch 库

### 6. 一行能搞定？
**简洁胜于繁琐**

- ❌ **过度封装**：为简单逻辑写 10 行辅助函数
- ✅ **正确做法**：如果逻辑足够简单，内联就好
- 💡 **示例**：不需要 `isEmpty(arr)` 函数，直接写 `arr.length === 0`

### 7. 写最小可工作代码
**只有到这一步才动手写**

- 只实现需求明确要求的功能
- 不添加"可能有用"的参数
- 不预设未来的扩展点
- 但要写得正确、清晰、可维护

## 永不妥协的底线

以下原则**绝对不能省略**，即使为了追求简洁：

### ✅ 必须保留

1. **安全性**
   - 输入验证和清理
   - SQL 参数化查询（防止注入）
   - XSS 防护
   - 认证和授权检查
   - 敏感数据加密

2. **错误处理**
   - 关键操作的 try-catch
   - 网络请求失败处理
   - 用户友好的错误提示
   - 日志记录（生产环境问题追踪）

3. **数据完整性**
   - 数据库事务
   - 数据丢失保护
   - 原子操作保证

4. **可访问性**（前端）
   - 语义化 HTML
   - ARIA 标签（必要时）
   - 键盘导航支持
   - 屏幕阅读器兼容

5. **关键边界条件**
   - 空值、null、undefined 检查
   - 数组越界保护
   - 除零检查

### ❌ 可以省略

- 过度抽象的接口层
- "万一需要"的配置选项
- 不必要的中间数据结构
- 过度细分的函数拆分
- 过早的性能优化

## 典型反模式

### 反模式 1：过度抽象
```typescript
// ❌ 过度设计
interface DataFormatter {
  format(data: any): string;
}
class CSVFormatter implements DataFormatter { /* 50 lines */ }
class JSONFormatter implements DataFormatter { /* 50 lines */ }
const formatter: DataFormatter = new CSVFormatter();

// ✅ 简单直接（只需要 CSV）
const csv = rows.map(r => r.join(',')).join('\n');
```

### 反模式 2：不必要的工具函数
```typescript
// ❌ 封装简单逻辑
function isEmptyArray(arr: any[]): boolean {
  return arr.length === 0;
}

// ✅ 直接使用
if (arr.length === 0) { /* ... */ }
```

### 反模式 3：重复造轮子
```typescript
// ❌ 自己实现已有功能
function debounce(fn: Function, delay: number) { /* 15 lines */ }

// ✅ 使用已安装的依赖（假设项目用 lodash）
import { debounce } from 'lodash';
```

### 反模式 4：过早引入依赖
```html
<!-- ❌ 安装复杂日期库 -->
<script src="flatpickr.js"></script>
<script>flatpickr('#date', { /* config */ });</script>

<!-- ✅ 使用原生特性 -->
<input type="date">
```

## 正确案例

### 案例 1：表单验证
```html
<!-- ❌ 引入 Formik + Yup -->
<Formik validationSchema={schema}>...</Formik>

<!-- ✅ 简单表单用 HTML5 -->
<input type="email" required pattern="[^@]+@[^@]+\.[^@]+">
<input type="number" min="0" max="100">
```

### 案例 2：数组去重
```typescript
// ❌ 安装工具库
import { uniq } from 'lodash';
const unique = uniq(arr);

// ✅ 使用标准特性
const unique = [...new Set(arr)];
```

### 案例 3：数据获取
```typescript
// ❌ 引入复杂状态管理（简单场景）
import { useQuery } from 'react-query';

// ✅ 简单场景直接 fetch
const [data, setData] = useState(null);
useEffect(() => {
  fetch('/api/data')
    .then(r => r.json())
    .then(setData)
    .catch(console.error); // ✅ 保留错误处理
}, []);
```

## 工作流程

### 开发新功能时
1. **理解需求** → 明确必须实现什么（不是可能需要什么）
2. **阅读现有代码** → 搜索类似功能、理解现有模式
3. **走决策阶梯** → 从第 1 级开始检查
4. **实现** → 在找到的阶梯层级停下，用最简方案
5. **质量检查** → 确保安全、错误处理、数据完整性没有遗漏

### 代码审查时
检查清单：
- [ ] 这个新代码是否真的需要？（YAGNI）
- [ ] 是否复用了现有代码？
- [ ] 是否优先使用了标准库/原生特性？
- [ ] 是否引入了不必要的依赖？
- [ ] 安全性是否到位？（输入验证、防注入、授权）
- [ ] 错误处理是否完整？
- [ ] 关键边界条件是否覆盖？

## 实施建议

### 对于团队
1. **代码审查关注点**：不仅看"功能是否实现"，更要问"是否可以更简单"
2. **依赖审查**：每次引入新依赖前问"现有方案是否足够"
3. **模式文档**：记录项目中可复用的模式和工具函数

### 对于 AI 辅助开发
1. **提示 AI**：在请求实现前，明确说"用最简方案，优先原生特性"
2. **审查 AI 代码**：AI 倾向过度设计，需要人工精简
3. **反馈循环**：发现 AI 过度设计时，指出并要求简化

### 关键心态
- **懒于方案，勤于阅读**：花时间理解现有代码和平台能力，节省写代码时间
- **代码是负债，不是资产**：每行代码都有维护成本
- **简单不等于简陋**：质量底线不能妥协

## 度量指标

### 健康信号
- 依赖数量保持稳定或减少
- 新功能代码行数趋势下降
- 代码复用率上升
- Bug 率下降（代码越少，出错空间越小）

### 警告信号
- 依赖数量快速增长
- 大量单用途工具函数
- 相似功能的不同实现
- 过度抽象导致理解困难
